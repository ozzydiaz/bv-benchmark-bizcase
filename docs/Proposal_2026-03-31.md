## 4. Calculation Chain

```
RVtools exports (vInfo, vHost tabs)
         │
         ▼
1-Client Variables        2a/2b/2c-Consumption Plans      Benchmark Assumptions
(~40 client inputs)       (10-yr Azure estimates × 3)     (65 overridable params)
         │                         │                               │
         └──────────────┬──────────┘───────────────────────────────┘
                        ▼
            Status Quo Estimation
            ──────────────────────
            CAPEX: servers, storage, NW, fitout
            OPEX: hardware maintenance, DC lease/power,
                  license costs, backup/DR, IT admin
                  (10-year, with growth compounding)
                        │
                        ▼
           Retained Costs Estimation
           ───────────────────────────
           On-prem costs that remain during/after migration
           Per workload × migration ramp-up schedule
           Virtualization, Win/SQL licenses declining with migration
                        │
                        ▼
            Depreciation Schedule
            ─────────────────────
            7-year lookback + 10-year forward
            Per asset class (servers, storage, NW+fitout)
            Per workload (3 parallel schedules)
                        │
                        ▼
          Detailed Financial Case
          ─────────────────────────────────────
          34 P&L line items × 11 years (Y0–Y10)
          Status Quo columns  ||  Azure columns
          CAPEX + OPEX tagged per category
                        │
                        ▼
          Summary Financial Case
          ──────────────────────
          NPV (10-yr & 5-yr), with and without terminal value
          ROI, Payback period
          Cash Flow delta table (status quo vs Azure)
          P&L view (10-yr & 5-yr)
          Charts: waterfall, cash flow, stacked bars
                        │
          ┌─────────────┼─────────────┐
          ▼             ▼             ▼
   Cash Flow       AVS Output    Transfo Value
   Output          (annual       Waterfall
   (detailed       run-rate,     (cost category
   on-prem vs      savings,      reduction
   Azure, by       MS invest.)   breakdown)
   cost type)
                              + IT Productivity
                              + Payback Analysis
                              + Net Interest Income
```

### Key Formula Patterns
| Pattern | Example |
|---|---|
| Ramp-up interpolation | `=AVERAGE(prior_ramp%, this_ramp%) × full_run_rate` |
| Retained cost decay | `base_cost × (1 - migration_ramp%)` with annual growth |
| Asset depreciation | `=MAX(0, annual_depr × (1 + growth_rate)^year)` |
| NPV discounting | `=CF / (1 + WACC)^year` |
| Terminal value | `=CF_year10 × (1 + g) / (WACC - g)` |
| CAGR | `=(end/start)^(1/n) - 1` |

## 6. Complexity Assessment

### ✅ High-Feasibility Areas
- **Pure arithmetic throughout** — no solver, no circular references, no array formulas requiring Excel engine
- **Linear dependency chain** — each sheet feeds forward only
- **Clean input/output separation** — all user inputs are in column D of `1-Client Variables` and in identified cells of the consumption plan sheets
- **Only one named range** (`WACC`) — trivially replaceable
- **Well-documented benchmarks** — survey sources, notes, prior values all present

### ⚠️ Medium-Complexity Areas
- **3-workload parallelism** — identical formula blocks replicated three times; maps well to a list/loop in code
- **Migration ramp-up interpolation** — piecewise linear across 10 years; straightforward in Python/NumPy
- **Depreciation lookback** — 7 years of pre-existing asset amortization; requires initializing a schedule from baseline
- **DC exit type branching** (Static vs Proportional) — two distinct calculation paths

### 🔴 Unknown / Risk Areas
| Risk | Detail |
|---|---|
| **VBA macros** | Sheet `VBA_Do_Not_Delete >>` is a placeholder; macros live in `.xlsm` VBA modules which aren't readable via openpyxl. Need to open in Excel to inspect. Could be trivial (button click) or non-trivial (data manipulation). |
| **Chart logic** | Slide 4/5, the waterfall and cash flow charts depend on specific data layouts; re-creating these programmatically requires a charting library decision. |
| **WCF Automated Price Tool** | License costs reference an external "WCF price tool" for level B/D lookup — needs replacement with a pricing API or static table. |
| **`Benchmark Assumptions` phantom columns** | `max_column = 16,374` but actual data ends at column 18. Phantom range is an Excel artifact; the usable data is only 18 columns wide. |

## 7. Recommended Architecture

### Recommended Path: Option B → Option A (Sequential, Not Parallel)

Options B and A are **not independent** — they are designed to be a **staged build** where B is Phase 1 and A is Phase 2. The calculation engine code written for B is **directly reused** in A with zero rework. See Section 9 for detail.

---

### Option A — Full Python Rewrite + AI Agent (Production)

```
┌──────────────────────────────────────────────────┐
```

## 9. Option A vs Option B — Relationship & Reuse

**Option B is Phase 1 of Option A — they share the same calculation engine.**

The central insight is that the only component that is _unique_ to each option is the **presentation layer**:

| Component | Option B (Streamlit) | Option A (Production) | Reused? |
|---|---|---|---|
| RVtools parser | ✅ Built in B | ✅ Reused in A | **100%** |
| Benchmark config store | ✅ Built in B | ✅ Reused in A | **100%** |
| Calculation engine (Python functions) | ✅ Built in B | ✅ Reused in A | **100%** |
| NPV / ROI / Payback engine | ✅ Built in B | ✅ Reused in A | **100%** |
| Streamlit UI | ✅ Built in B | ❌ Replaced | Throwaway |
| Plotly chart wrappers in Streamlit | ✅ Built in B | ~Partially reused | Partial |
| FastAPI service layer | ❌ Not in B | ✅ Built in A | New in A |
| AI agent / tool definitions | ❌ Not in B | ✅ Built in A | New in A |
| React/Next.js front-end | ❌ Not in B | ✅ Built in A | New in A |

**Conclusion:** Moving from B to A does **not** make Option B throwaway in any meaningful sense.  
~70–80% of the total engineering work (the calculation engine) is shared. The Streamlit UI layer — roughly 1–2 weeks of effort — is the only part that gets superseded. That UI layer delivers value during B as a demo/validation tool, so it earns its keep before being replaced.

**Recommended path:** Build Option B first. Use it to validate the calculation engine against the real workbook with real client data. Then promote the calculation engine into a FastAPI service and build the AI agent on top of it in Option A.

## 10. Summary Verdict

**Feasibility: High.** The workbook is complex in scope but architecturally straightforward — it is 100% deterministic arithmetic with no exotic Excel dependencies. The calculation chain is linear and well-separated. The ~40 primary inputs and ~65 benchmark parameters are cleanly identified.

The highest-value AI capability is at the **intake layer**: letting an agent converse with the field team, parse RVtools files, ask clarifying questions about workloads and license inventory, and populate the inputs — then run the math instantly rather than manually filling the spreadsheet. The model also opens naturally to **scenario comparison** (e.g. "show 2-year vs 3-year migration ramp") and **sensitivity analysis** (e.g. "what happens to NPV if the VM-to-server ratio is 8 instead of 12").

The one unknown that needs resolution before committing to scope is the **VBA macro content** — this requires opening the file in Excel with macro inspection enabled.

### Follow-up Q&A

---

**Q: Is Option A inclusive of Option B, or are they independent paths?**

They are **staged, not parallel**. Option B is Phase 1 and Option A is Phase 2 of the same build. The only thing that differs between them is the presentation layer — Streamlit vs FastAPI + React + AI agent. The entire calculation engine (RVtools parser, benchmark store, all financial math) is written once in B and carried forward verbatim into A.

---

**Q: If Option B is built first, does moving to Option A make Option B throwaway?**

Mostly no. Roughly **70–80% of total effort** — everything in the calculation engine — transfers 100%. The only piece that gets superseded is the Streamlit UI, which is approximately 1–2 weeks of work. That UI is not waste: it serves as the validation harness for the calculation engine against real workbook data *before* investing in production infrastructure. You would want to build it as a testbed regardless.

**Recommended sequence:** Build B → validate math against the real workbook with real client data → promote the engine to FastAPI → layer the AI agent on top in Option A.

#### Phase 2 — Calculation Engine (Weeks 2–4)

Build and unit-test each module in dependency order, validating each layer's outputs against the real workbook before moving to the next:

1. `rvtools_parser.py` — parse vInfo/vHost, return a typed `WorkloadInventory` object
2. `benchmarks.py` — load defaults from YAML, accept per-session overrides
3. `status_quo.py` — replicate `Status Quo Estimation` sheet (CAPEX + OPEX, 10-year)
4. `depreciation.py` — replicate `Depreciation Schedule` (7-year lookback + 10-year forward)
5. `retained_costs.py` — replicate `Retained Costs Estimation` (ramp-down per workload)
6. `financial_case.py` — produce the 34-row × 11-column P&L matrix
7. `outputs.py` — NPV, 5-yr/10-yr, ROI, payback, waterfall breakdowns

**Validation gate:** Each module's numeric output must match the reference workbook (with the same inputs) to within rounding tolerance before proceeding.

#### Phase 3 — Streamlit UI (Weeks 4–5)

- Multi-page app with RVtools file uploader, client variable form, benchmark override sidebar
- Consumption plan entry (migration ramp % by year, Azure annual spend anchor)
- Results page: Plotly waterfall, cash flow chart, NPV/ROI summary card
- Optional: thin OpenAI chat panel on the intake page for guided data collection
